# Container and Cloud Security Tools

> Reference guide for the tools used in the Container and Cloud Security Blueprint, organized by security layer.

## Overview

This document covers the tools referenced in the [Container and Cloud Security Blueprint](../best-practices/container-and-cloud-security-blueprint.md). Each tool is described with its purpose, primary use case, key commands, and integration guidance. Tools are grouped by the security layer they address.

## Layer 1 - Container Image Security

### Trivy

> Universal vulnerability, secret, and misconfiguration scanner for container images, filesystems, git repositories, and IaC.

**When to use:** Scan images before push, scan IaC definitions in pull requests, and run scheduled scans against already-deployed images.

**Key features:**
- Scans container images, Dockerfiles, Terraform, Helm, and Kubernetes manifests in a single tool
- Detects CVEs, exposed secrets, and IaC misconfigurations
- Outputs SARIF, JSON, and table formats for CI integration
- Supports SBOM generation in CycloneDX and SPDX formats

**Usage examples:**

```bash
# Scan a container image - fail on critical CVEs with an available fix
trivy image --severity CRITICAL --ignore-unfixed myregistry/api:latest

# Generate an SBOM in CycloneDX format
trivy image --format cyclonedx --output sbom.json myregistry/api:latest

# Scan a Terraform directory
trivy config ./infra/terraform/

# Scan a Kubernetes manifest
trivy config ./k8s/deployment.yaml
```

**CI integration (GitHub Actions):**

```yaml
- name: Scan image with Trivy
  # Pin third-party actions to a reviewed commit SHA, not a mutable branch or tag.
  uses: aquasecurity/trivy-action@<commit-sha>
  with:
    image-ref: myregistry/api:${{ github.sha }}
    format: sarif
    output: trivy-results.sarif
    severity: CRITICAL,HIGH
    ignore-unfixed: true
```

**Alternatives:** Grype (image CVEs only, faster), Snyk Container, Docker Scout

**Resources:**
- [Trivy Documentation](https://aquasecurity.github.io/trivy/)
- [GitHub Repository](https://github.com/aquasecurity/trivy)

### Grype

> Fast container image and filesystem vulnerability scanner by Anchore.

**When to use:** When speed is a priority for image scanning in CI; complements Trivy for cross-validation of CVE findings.

**Key features:**
- High-speed CVE scanning against image layers and filesystems
- Integrates with Syft for SBOM-first scanning workflows
- Supports OCI images, Docker archives, and directories
- Outputs JSON and table formats

**Usage examples:**

```bash
# Scan an image
grype myregistry/api:latest

# Fail only on critical findings
grype myregistry/api:latest --fail-on critical

# Scan from an SBOM generated by Syft
syft myregistry/api:latest -o json | grype
```

**Alternatives:** Trivy (broader scope including IaC and secrets), Snyk Container

**Resources:**
- [Grype GitHub Repository](https://github.com/anchore/grype)
- [Anchore Documentation](https://docs.anchore.com/)

### Syft

> SBOM (Software Bill of Materials) generator for container images, filesystems, and OCI archives, by Anchore.

**When to use:** Generate an SBOM at build time to create a persistent, queryable inventory of every package in an image. Feed the SBOM directly into Grype for vulnerability scanning, or attach it to the image via Cosign as a signed attestation.

**Key features:**
- Generates SBOMs in SPDX, CycloneDX, and Syft JSON formats
- Works with OCI images, Docker archives, directories, and filesystems
- Integrates natively with Grype for SBOM-first scanning workflows
- SBOM can be attached to the image registry as a signed attestation via Cosign

**Usage examples:**

```bash
# Generate a CycloneDX SBOM for an image
syft myregistry/api:latest -o cyclonedx-json > sbom.cyclonedx.json

# Generate a Syft JSON SBOM and pipe directly into Grype
syft myregistry/api:latest -o json | grype

# Generate an SPDX SBOM for a local directory
syft dir:./app -o spdx-json > sbom.spdx.json

# Attach the SBOM as a signed attestation (requires Cosign)
cosign attest --predicate sbom.cyclonedx.json --type cyclonedx myregistry/api@sha256:abc123...
```

**Alternatives:** Trivy (`--format cyclonedx`), Docker Scout SBOM

**Resources:**
- [Syft GitHub Repository](https://github.com/anchore/syft)
- [Anchore Documentation](https://docs.anchore.com/)

### Hadolint

> Dockerfile linter that enforces best practices and catches misconfigurations before the image is built.

**When to use:** Lint every Dockerfile in the pull request gate; catches issues such as unpinned base images, missing USER instruction, and insecure `apt-get` patterns before build time.

**Key features:**
- Rules aligned with CIS Docker Benchmark and Dockerfile best practices
- Integrates with shellcheck to lint inline shell commands in RUN instructions
- Outputs SARIF and JSON for CI integration
- Rules can be suppressed inline or via a config file

**Usage examples:**

```bash
# Lint a Dockerfile
hadolint Dockerfile

# Output JSON for CI
hadolint -f json Dockerfile

# Suppress a specific rule
hadolint --ignore DL3008 Dockerfile

# Run without local install via Docker
docker run --rm -i hadolint/hadolint < Dockerfile
```

**Configuration (`.hadolint.yaml`):**

```yaml
ignore:
  - DL3008  # pin versions in apt-get install
failure-threshold: warning
```

**Alternatives:** Checkov (broader IaC scope), Dockerfile linting within Trivy

**Resources:**
- [Hadolint GitHub Repository](https://github.com/hadolint/hadolint)
- [Hadolint Rules Reference](https://github.com/hadolint/hadolint/wiki/DL3001)

### Cosign

> Container image signing and verification tool from the Sigstore project.

**When to use:** Sign every image digest after a successful build; verify signatures at admission time to prevent unsigned or tampered images from reaching the cluster.

**Key features:**
- Signs image digests (not mutable tags) using keyless signing via the Sigstore Fulcio certificate authority
- Attaches signatures, SBOMs, and attestations as OCI artifacts alongside the image in the registry
- Integrates natively with Kyverno and OPA Gatekeeper for admission-time signature verification
- Supports hardware key stores and cloud KMS (AWS KMS, GCP KMS, HashiCorp Vault)

**Usage examples:**

```bash
# Sign an image digest (keyless - uses OIDC token from CI environment)
cosign sign myregistry/api@sha256:abc123...

# Sign with a KMS key
cosign sign --key awskms:///arn:aws:kms:region:account:key/key-id myregistry/api@sha256:abc123...

# Verify a signature produced by GitHub Actions keyless signing
cosign verify \
  --certificate-identity-regexp "https://github.com/org/repo/.github/workflows/.*" \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com \
  myregistry/api@sha256:abc123...

# Attach an SBOM as a signed attestation
cosign attest --predicate sbom.json --type cyclonedx myregistry/api@sha256:abc123...
```

**Alternatives:** Notary v2 (CNCF standard, heavier setup), Docker Content Trust (legacy, uses Notary v1)

**Resources:**
- [Cosign Documentation](https://docs.sigstore.dev/cosign/signing/overview/)
- [Sigstore GitHub](https://github.com/sigstore/cosign)

## Layer 2 - Kubernetes Runtime Security

### kube-bench

> Checks Kubernetes cluster configuration against the CIS Kubernetes Benchmark.

**When to use:** Run against every cluster after initial setup and after upgrades to validate baseline security configuration.

**Key features:**
- Covers all CIS Kubernetes Benchmark sections: API server, etcd, scheduler, controller manager, kubelet, and RBAC
- Can run as a Kubernetes Job (inside the cluster) or as a binary directly on a node
- Outputs pass/fail results with remediation guidance for each failed check

**Usage:**

```bash
# Run as a Docker container on a node (read-only host access)
docker run --rm --pid=host \
  -v /etc:/etc:ro \
  -v /var:/var:ro \
  aquasec/kube-bench:latest

# Run as a Kubernetes Job
kubectl apply -f https://raw.githubusercontent.com/aquasecurity/kube-bench/main/job.yaml
kubectl logs -l app=kube-bench
```

**Alternatives:** kubescape (broader scope, includes NSA/CISA guidance), Polaris

**Resources:**

### Kyverno

> Kubernetes-native policy engine for admission control, mutation, and generation of resources.

**When to use:** Enforce workload security standards (non-root, no privileged containers, image signature verification) at admission time without writing Rego; generate default NetworkPolicies or resource quotas for new namespaces automatically.

**Key features:**
- Policies written as Kubernetes YAML - no external policy language required
- Three policy modes: validate (block or audit), mutate (patch resources), generate (create dependent resources)
- Verifies Cosign image signatures natively within a ClusterPolicy
- Generates policy reports for violations across already-admitted resources

**Example policy - require non-root:**

```yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: require-run-as-non-root
spec:
  rules:
    - name: check-runAsNonRoot
      match:
        any:
          - resources:
              kinds: [Pod]
      validate:
        failureAction: Enforce
        message: "Containers must run as non-root."
        pattern:
          spec:
            containers:
              - securityContext:
                  runAsNonRoot: true
```

**Example policy - verify image signature:**

```yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: verify-image-signature
spec:
  rules:
    - name: check-signature
      match:
        any:
          - resources:
              kinds: [Pod]
      verifyImages:
        - imageReferences: ["myregistry/*"]
          failureAction: Enforce
          attestors:
            - entries:
                - keyless:
                    issuerRegExp: https://token\.actions\.githubusercontent\.com
                    subjectRegExp: https://github\.com/org/repo/.+
                    rekor:
                      url: https://rekor.sigstore.dev
```

**Alternatives:** OPA Gatekeeper (Rego-based, more flexible for complex logic, steeper learning curve), Pod Security Admission (built-in, limited to PSS levels only)

**Resources:**
- [Kyverno Documentation](https://kyverno.io/docs/)
- [Kyverno Policy Library](https://kyverno.io/policies/)

### OPA Gatekeeper

> Open Policy Agent admission controller for Kubernetes, enforcing custom policies written in Rego.

**When to use:** Enforce complex, organization-specific policies that require logic beyond Kyverno's pattern matching - such as cross-field validation, external data lookups via the OPA cache, or constraints spanning multiple resource types.

**Key features:**
- Policies written in Rego (OPA's declarative policy language)
- ConstraintTemplates define reusable policy schemas; Constraint resources instantiate them per cluster with specific parameters
- Audit mode reports existing violations without blocking new admissions, allowing gradual rollout
- Syncs Kubernetes resources into the OPA data document for cross-resource policy evaluation

**Example - block privileged containers:**

```yaml
# ConstraintTemplate
apiVersion: templates.gatekeeper.sh/v1
kind: ConstraintTemplate
metadata:
  name: k8sdenyprivileged
spec:
  crd:
    spec:
      names:
        kind: K8sDenyPrivileged
  targets:
    - target: admission.k8s.gatekeeper.sh
      rego: |
        package k8sdenyprivileged
        violation[{"msg": msg}] {
          c := input.review.object.spec.containers[_]
          c.securityContext.privileged == true
          msg := sprintf("Container %v must not run privileged", [c.name])
        }
---
# Constraint instance
apiVersion: constraints.gatekeeper.sh/v1beta1
kind: K8sDenyPrivileged
metadata:
  name: deny-privileged
spec:
  match:
    kinds:
      - apiGroups: [""]
        kinds: ["Pod"]
```

**Alternatives:** Kyverno (YAML-native, lower barrier to entry), Pod Security Admission (built-in, limited to PSS levels)

**Resources:**
- [OPA Gatekeeper Documentation](https://open-policy-agent.github.io/gatekeeper/)
- [OPA Documentation](https://www.openpolicyagent.org/docs/)

### Falco

> Cloud-native runtime security tool for detecting unexpected behavior in containers and hosts using eBPF.

**When to use:** Detect runtime anomalies after workloads are admitted - unexpected process execution inside a container, writes to sensitive paths, unexpected outbound network connections, or privilege escalation attempts.

**Key features:**
- Uses eBPF (or a kernel module) to capture system call events with low overhead
- Rules written in YAML with a condition/output/priority model; no code compilation required
- Ships alerts to stdout, syslog, webhooks, or SIEM via Falcosidekick
- Ships a maintained rule library covering MITRE ATT&CK for Containers techniques

**Example rules:**

```yaml
# Detect a shell spawned interactively inside a container
- rule: Terminal Shell in Container
  desc: A shell was spawned in a container with a terminal attached
  condition: >
    spawned_process and container and shell_procs and proc.tty != 0
  output: >
    Shell spawned in container (user=%user.name container=%container.name
    image=%container.image.repository command=%proc.cmdline)
  priority: WARNING

# Detect writes to /etc inside a container
- rule: Write to /etc in Container
  desc: Unexpected write to the /etc directory inside a container
  condition: >
    open_write and container and fd.name startswith /etc
  output: >
    Write to /etc in container (file=%fd.name container=%container.name
    image=%container.image.repository)
  priority: ERROR
```

**Alternatives:** Tetragon (Cilium, eBPF-native with enforcement capabilities), Sysdig

**Resources:**
- [Falco Documentation](https://falco.org/docs/)
- [Falco Rules Repository](https://github.com/falcosecurity/rules)
- [Falcosidekick (output routing)](https://github.com/falcosecurity/falcosidekick)

## Layer 3 - Cloud IAM

### AWS IAM Access Analyzer

> AWS-managed service that identifies IAM roles, policies, and resources with unused or over-broad permissions.

**When to use:** Audit IAM permissions continuously; generate least-privilege policies from CloudTrail activity; detect unintended external or public access to S3 buckets, KMS keys, SQS queues, and other resource types.

**Key features:**
- Analyzes resource-based policies for cross-account and public access, surfacing findings with the affected principal and access level
- Generates least-privilege IAM policies based on actual CloudTrail access activity over a configurable time window
- Validates IAM policy documents for syntax errors and best-practice violations before they are attached
- Reports unused IAM roles, access keys older than a threshold, and unused passwords

**Usage (AWS CLI):**

```bash
# List external-access findings for an analyzer
aws accessanalyzer list-findings \
  --analyzer-arn arn:aws:access-analyzer:region:account:analyzer/name

# Validate a policy document for errors and warnings
aws accessanalyzer validate-policy \
  --policy-type IDENTITY_POLICY \
  --policy-document file://policy.json

# Generate a service last-accessed report for a role
aws iam generate-service-last-accessed-details \
  --arn arn:aws:iam::account:role/MyRole
```

**Resources:**
- [AWS IAM Access Analyzer Documentation](https://docs.aws.amazon.com/IAM/latest/UserGuide/what-is-access-analyzer.html)

### Azure Privileged Identity Management (PIM) / Entra ID

> Microsoft's managed service for just-in-time privileged access, access reviews, and least-privilege enforcement across Azure and Microsoft 365.

**When to use:** Audit and govern Azure RBAC role assignments; enforce just-in-time (JIT) activation for privileged roles; run access reviews to detect unused or over-permissive assignments across subscriptions and resource groups.

## Layer 4 - Infrastructure as Code

### Checkov

> Static analysis tool for IaC that detects security misconfigurations in Terraform, CloudFormation, Kubernetes, Helm, Dockerfiles, and ARM templates.

**When to use:** Scan all IaC files in pull requests; integrates with most CI platforms and outputs SARIF for GitHub Advanced Security code scanning.

**Key features:**
- 1,000+ built-in checks mapped to CIS, NIST, PCI DSS, HIPAA, SOC 2, and GDPR requirements
- Supports custom checks written in Python or YAML
- Graph-based analysis for Terraform to evaluate resource relationships (e.g., security group attached to a public subnet)
- Suppression via inline comments or a centralized config file with required justification

**Usage examples:**

```bash
# Scan a Terraform directory
checkov -d ./infra/terraform/ --framework terraform

# Scan Kubernetes manifests
checkov -d ./k8s/ --framework kubernetes

# Output SARIF for GitHub Advanced Security
checkov -d ./infra/ --output sarif --output-file-path ./results/

# Skip specific checks with justification
checkov -d ./infra/ --skip-check CKV_AWS_21,CKV_AWS_18
```

**CI integration (GitHub Actions):**

```yaml
- name: Checkov IaC Scan
  # Pin third-party actions to a reviewed commit SHA, not a mutable branch or tag.
  uses: bridgecrewio/checkov-action@<commit-sha>
  with:
    directory: infra/
    framework: terraform
    output_format: sarif
    output_file_path: checkov-results.sarif
    soft_fail: false
```

**Alternatives:** Trivy config scanning (unified image + IaC scanner), KICS (broad format support with Rego queries), tfsec (Terraform-only, now merged into Trivy)

**Resources:**
- [Checkov Documentation](https://www.checkov.io/)
- [Checkov GitHub Repository](https://github.com/bridgecrewio/checkov)

### KICS (Keeping Infrastructure as Code Secure)

> Open source IaC security scanner by Checkmarx with broad format support and Rego-based custom queries.

**When to use:** When broad IaC format coverage is needed across Terraform, CloudFormation, Kubernetes, Helm, Ansible, and Docker Compose in a single tool, or when custom queries are required using OPA Rego.

**Key features:**
- 2,000+ queries across all major IaC formats
- Queries written in Rego - extensible without rebuilding the tool

### Conftest (OPA)

> Policy testing tool for structured configuration files using OPA Rego policies; used alongside IaC scanners to enforce organization-specific rules.

**When to use:** Enforce custom organizational policies not covered by Checkov or KICS - such as mandatory resource tagging, approved image registry allowlists, naming conventions, or cross-resource constraints evaluated against `terraform plan` output.

**Key features:**
- Tests any structured data format: JSON, YAML, HCL, TOML, XML, Dockerfile, and Kubernetes manifests
- Evaluates against `terraform show -json` output to test the plan rather than the source
- Supports pulling shared policy bundles from OCI registries
- Integrates into CI without requiring a running OPA server

**Usage examples:**

```bash
# Test a Terraform plan
terraform show -json tfplan.binary | conftest test -

# Test Kubernetes manifests
conftest test k8s/deployment.yaml

# Pull and use a remote policy bundle
conftest pull oci://ghcr.io/org/policies:latest
conftest test k8s/
```

**Example policy - deny public S3 bucket:**

```rego
package main

deny[msg] {
  input.resource_type == "aws_s3_bucket"
  input.config.acl == "public-read"
  msg := sprintf("S3 bucket %v must not be public-read", [input.address])
}
```

**Alternatives:** Kyverno (Kubernetes-native, YAML-based), OPA Gatekeeper (admission-time enforcement in cluster)

**Resources:**
- [Conftest Documentation](https://www.conftest.dev/)
- [OPA Documentation](https://www.openpolicyagent.org/docs/)

## Layer 5 - Cloud Security Posture Management

### AWS Security Hub

> Centralized CSPM service for AWS that aggregates findings from GuardDuty, Inspector, Macie, IAM Access Analyzer, and third-party tools.

**When to use:** Continuous posture monitoring across AWS accounts; detecting drift from IaC-defined state; aggregating findings from multiple AWS security services into a single normalized view.

**Key features:**
- Automated security checks against CIS AWS Foundations Benchmark, AWS Foundational Security Best Practices standard, and PCI DSS
- Aggregates findings across multiple AWS security services and certified partner integrations
- Cross-account and cross-region aggregation via AWS Organizations for consolidated visibility
- Supports the ASFF (Amazon Security Finding Format) for custom finding ingestion from external tools

**Usage (AWS CLI):**

```bash
# List active CRITICAL and HIGH findings across all controls
aws securityhub get-findings \
  --filters '{"SeverityLabel":[{"Value":"CRITICAL","Comparison":"EQUALS"},{"Value":"HIGH","Comparison":"EQUALS"}],"RecordState":[{"Value":"ACTIVE","Comparison":"EQUALS"}]}' \
  --query 'Findings[*].{Title:Title,Severity:Severity.Label,Resource:Resources[0].Id}' \
  --output table

# List enabled security standards and their subscription ARNs
aws securityhub get-enabled-standards --output table

# List failed controls for a specific standard (use ARN from the command above)
aws securityhub describe-standards-controls \
  --standards-subscription-arn <standards-subscription-arn> \
  --control-status FAILED \
  --output table
```

**Resources:**
- [AWS Security Hub Documentation](https://docs.aws.amazon.com/securityhub/)

## Sandboxed Runtimes

### gVisor

> User-space kernel that intercepts and re-implements Linux system calls in user space, providing stronger isolation than runC at the cost of I/O and network performance.

**When to use:** High-risk or multi-tenant workloads where isolation matters more than throughput. Prefer the KVM platform where available; the ptrace platform imposes ~95% syscall overhead and collapses I/O under concurrency.

**Performance characteristics (Wang et al. 2022):**
- Syscall overhead: ~50% (KVM driver) / ~95% (ptrace driver)
- Network throughput: ~655 KB/s vs ~96,000 KB/s bare metal (-99%)
- Memory per container: ~8 MB
- Startup overhead: +0.14 s vs runC

**Kubernetes RuntimeClass:**

```yaml
apiVersion: node.k8s.io/v1
kind: RuntimeClass
metadata:
  name: gvisor
handler: runsc
---
# Reference in Pod spec
spec:
  runtimeClassName: gvisor
```

**Resources:**
- [gVisor Documentation](https://gvisor.dev/docs/)
- [gVisor GitHub Repository](https://github.com/google/gvisor)

---

### Kata Containers

> Lightweight VM-based container runtime providing hardware-level isolation with near-runC performance for CPU, network, and syscall throughput.

**When to use:** When hardware isolation is required but CPU, network throughput, and syscall-heavy workloads cannot tolerate gVisor's overhead. Avoid deploying above ~40 simultaneous containers on an 8 GB host - memory pressure (~110 MB per container) causes startup time to spike significantly.

**Performance characteristics (Wang et al. 2022):**
- Syscall overhead: <1%
- Network throughput: ~95,600 KB/s (near runC)
- Sequential I/O overhead: 12-17%
- Memory per container: ~110 MB
- Startup overhead: +0.44 s vs runC

**Kubernetes RuntimeClass:**

```yaml
apiVersion: node.k8s.io/v1
kind: RuntimeClass
metadata:
  name: kata
handler: kata
---
# Reference in Pod spec
spec:
  runtimeClassName: kata
```
